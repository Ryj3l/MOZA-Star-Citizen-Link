# T-26: Deep replay sensors

- **Task ID:** T-26
- **Title:** Implement `ReplayLogSensor`, `ReplayAudioSensor`, `ReplayScreenSensor` as `ISensor` implementations that feed the real event bus from a bundle. Extend the light CLI to support deep replay through the full pipeline.
- **Depends on:** T-10 (event bus), T-12 (fusion engine), T-14 (effect resolver), T-15 (safety limiter), T-25 (light CLI)
- **Phase:** 2 (after live audio/screen sensors exist) — but technically buildable as soon as the dependencies merge
- **Estimated effort:** L (1–3 days)
- **Branch:** `feat/26-replay-sensors-deep`

## Context

T-25 produced a light CLI that runs Game.log through the pattern library and emits events. T-26 closes the loop: replay drives the **full sensor stack** through the event bus, fusion engine, resolver, and safety limiter, producing both `events.jsonl` (sensor-level events) and `commands.jsonl` (the final force commands that would have reached the device). This lets a developer answer "what would the player have felt during this play session, given my current catalog and rule configuration?"

The replay sensors implement `ISensor` (PRP §5.3) and emit `SensorEvent`s to the event bus at the bundle's recorded wall-clock offsets, scaled to a configurable replay speed (default 1×). Everything downstream of the bus runs unchanged — same fusion rules, same gain stack, same safety limiter. The only difference from live operation is the source of the sensor events.

T-26 also finishes the bundle format support T-25 stubbed: audio feature ingestion, screen feature ingestion, and full validation rules per `docs/replay-bundle-schema.md`.

## Deliverables

1. **`src/Moza.ScLink.Replay.Sensors/`** — new .NET 8 class library under `src/`. References `Moza.ScLink.Core`. Csproj sets `<IsPackable>false</IsPackable>` and the App project explicitly excludes it from Release configuration project references.
2. **`src/Moza.ScLink.Replay.Sensors/ReplayClock.cs`** — `IReplayClock` abstraction wrapping the bundle's time base. Methods: `Start()`, `Stop()`, `AdvanceTo(TimeSpan)`, `Now` property, `SpeedMultiplier`. Test-substitutable (mock for unit tests).
3. **`src/Moza.ScLink.Replay.Sensors/ReplayLogSensor.cs`** — implements `ISensor`. On `StartAsync`, opens the bundle's `game.log`, iterates lines, computes each line's wall-clock offset (Game.log timestamps are absolute; subtract the recording start to get bundle-relative ms), and emits `SensorEvent` instances at those offsets according to the replay clock. Uses the same `StarCitizenEventParser` instance as the live `LogSensor`.
4. **`src/Moza.ScLink.Replay.Sensors/ReplayAudioSensor.cs`** — implements `ISensor`. Reads `audio-features.jsonl`, emits one `SensorEvent` per FFT frame (with `Kind = SensorKind.Audio`, `Features` populated from the per-band power values).
5. **`src/Moza.ScLink.Replay.Sensors/ReplayScreenSensor.cs`** — implements `ISensor`. Reads `screen-features.jsonl`, emits one `SensorEvent` per frame (with `Kind = SensorKind.Screen`, `Features` populated from the per-ROI feature vector).
6. **`src/Moza.ScLink.Replay.Sensors/ReplaySensorHost.cs`** — DI-friendly composite that wires the three sensors against a shared `IReplayClock`, exposes a single `StartAsync(bundle, ct)` entry point, and shuts down cleanly when the bundle's end is reached.
7. **`tools/Moza.ScLink.Replay/` CLI extensions** — extend Program.cs with a `--deep` flag on `replay`. When `--deep` is supplied:
   - Build a DI container with: `IEventBus`, `LogSensor` replaced by `ReplayLogSensor`, `AudioSensor` (if exists in your Phase 2 build) replaced by `ReplayAudioSensor`, `ScreenSensor` replaced by `ReplayScreenSensor`, the real `FusionEngine`, `EffectResolver`, `SafetyLimiter`.
   - Wire the output sink: instead of an `IForceFeedbackDevice`, register a `RecordingForceFeedbackDevice` that captures every `ForceCommand` to a `commands.jsonl` stream.
   - Run until the replay clock signals end-of-bundle, then flush and exit.
   - Write both `events.jsonl` (from a tee on the event bus) and `commands.jsonl` to the output directory.
8. **`tools/Moza.ScLink.Replay/Output/RecordingForceFeedbackDevice.cs`** — implements `IForceFeedbackDevice`, captures all commands to a stream, never touches a real device.
9. **`tools/Moza.ScLink.Replay/Bundle/AudioStripper.cs`** — extracts `audio-features.jsonl` from `audio.wav` using the FFT parameters in the manifest. Runs as part of the `strip` subcommand. Uses MathNet.Numerics (Phase 2 dependency; DDR already in scope if Phase 2 has started).
10. **`tools/Moza.ScLink.Replay/Bundle/ScreenStripper.cs`** — extracts `screen-features.jsonl` from `screen-rois.mp4` per ROI. Runs as part of the `strip` subcommand. Implementation note: for the first pass, accept that the screen-features file may already be present in the rich bundle (the live recorder emits both raw video and features); the stripper just verifies and copies. A future task can rebuild features from the video if needed.
11. **Full bundle validation** — T-25's validator becomes strict about audio/screen feature line counts now that those files are first-class.
12. **Test fixtures** — extend `tools/Moza.ScLink.Replay.Tests/Fixtures/sample-bundle/` with hand-crafted (or recorder-produced, if the Phase 2 recorder ships first) audio-features.jsonl and screen-features.jsonl files containing at least one event each that should trigger a fusion rule firing.
13. **Integration tests** — `ReplayPipelineTests.SampleBundle_Produces_Expected_Commands` loads the fixture, runs deep replay, asserts the `commands.jsonl` output contains the expected `PlayEffectCommand`s in the expected order. This is the regression test that protects the entire pipeline against drift.
14. **CLI documentation** — `tools/Moza.ScLink.Replay/README.md` updated with deep-replay examples.

## Non-goals

- No live recorder. Recorder is a separate Phase 2 task that produces bundles; T-26 only consumes them.
- No `--with-device` flag. Driving real hardware from a replay requires safety guards (allowlist check, master-gain cap, explicit `--i-have-a-clear-bench` flag). Future task after T-26 is stable.
- No `--speed 4x` faster-than-real-time. Replay runs at 1× by default. A speed flag is a future quality-of-life feature, not part of this task's contract.
- No replay-driven CI regression. Once T-26 lands and a representative fixture bundle is checked into the test project, **then** a follow-up task can wire a `Replay.Sensors.Tests` job into `.github/workflows/ci.yml`.
- No replay UI in the App. Replay is a CLI workflow; the App stays user-facing only.

## Implementation notes

- The replay clock must be **monotonic** within a bundle. Don't use `DateTimeOffset.Now` anywhere in the replay path — bundle offsets are absolute milliseconds from `wallClockStartUtc`.
- The fusion engine's evidence windows (PRP §6, T-12) are time-based. Replay must use the replay clock's time, not real time, when computing whether evidence is fresh. This is the trickiest integration point; budget time for it.
- The "tee" on the event bus is a diagnostic-mode-only feature. Use the existing `Moza.ScLink.Diagnostics.EventBusTap` (or equivalent — T-18 introduces a diagnostics panel that already taps the bus) so you don't need new bus instrumentation.
- Audio feature line schema: the band-power floats. Sensible: 8 logarithmically spaced bands 20 Hz – 16 kHz. Lock the schema in T-26 and update `docs/replay-bundle-schema.md` § `sources.audio.featuresPath` with the final field list. Bundles recorded before this lock are advisory.
- Screen feature line schema: per ROI, a small typed dictionary `{meanLuminance, edgeDensity, dominantColorHsv, ...}`. Same lock-and-document discipline as audio.
- For `RecordingForceFeedbackDevice`, capture the command **after** the safety limiter applies. The `commands.jsonl` is what would have hit the device.
- Output file format for `commands.jsonl`:
  ```json
  {"tMs": 12500, "command": "PlayEffect", "effectId": "quantum-spool-v1", "finalIntensity": 0.42, "durationMs": 8000, "frequencyHz": 34, "stateKey": "quantum-spool"}
  {"tMs": 20503, "command": "StopEffect", "stateKey": "quantum-spool"}
  ```
  One line per command. JSONL because diffs are clean and it streams.
- If T-26 ships before the Phase 2 recorder ships, hand-assembled fixture bundles will need hand-written audio-features.jsonl files. That's fine for the test fixtures; the schema is simple enough to author manually.

## Operator note

After T-26 merges, the workflow for tuning anything in the pipeline becomes:

1. Operator imports a bundle into `tools/recordings/`.
2. `dotnet run --project tools/Moza.ScLink.Replay -- replay tools/recordings/<bundle> --deep`.
3. Inspect `commands.jsonl` — these are what the wheel would have felt.
4. Edit any of: `Moza.ScLink.Logs/Patterns/v0.json`, `phase1-effect-catalog.json`, `phase1-rules.json`, gain settings, fusion rule weights.
5. Re-run step 2. Diff the two `commands.jsonl` files (`git diff --no-index old.jsonl new.jsonl`).
6. When the diff looks right, commit. Optionally re-run T-26's `ReplayPipelineTests` to confirm the fixture still passes.

This is the workflow that makes Phase 2 catalog tuning tractable. Without it, every change requires booting Star Citizen, plugging in a wheel, and replaying the exact scenario by hand. With it, the catalog becomes a tight inner-loop iteration.
