# T-13: Phase 1 effect catalog JSON

- **Task ID:** T-13
- **Title:** Load the Phase 1 effect catalog from JSON; remove hard-coded effect parameters from controller code
- **Depends on:** T-06
- **Phase:** 1
- **Estimated effort:** S (1–2 hours)
- **Branch:** `feat/13-effect-catalog`

## Context

The existing `ForceFeedbackController` hard-codes effect parameters. PRP §9.1 mandates moving these into `Moza.ScLink.Effects/Catalogs/phase1.json`. The starter bundle provides this file.

## Deliverables

1. `Moza.ScLink.Effects/Catalogs/phase1.json` placed in the project, marked `CopyToOutputDirectory="PreserveNewest"`.
2. `Moza.ScLink.Effects/EffectCatalog.cs` loading the JSON, with hot-reload via `FileSystemWatcher` and debounce.
3. `Moza.ScLink.Effects/Catalogs/EffectCatalogSchema.cs` deserialization types matching the JSON shape.
4. Schema versioning: if `schemaVersion` ≠ 1, log a warning and load defaults.
5. Schema validation: every effect must have valid `effectId`, `category`, `effectType`, and parameter ranges (intensity 0..1, frequency ≥ 0, duration ≥ 0).
6. Unit tests:
   - Happy path load
   - Missing schemaVersion → defaults
   - Out-of-range intensity → rejected with warning
   - Hot reload picks up changes within 1 second
   - Malformed JSON → defaults loaded
7. `EffectResolver` (added in T-14) consumes `EffectCatalog` rather than hard-coded parameters.

## Acceptance criteria

- [ ] Phase 1 catalog file present with all 7 effects from the starter bundle.
- [ ] Catalog loads successfully on app start.
- [ ] All 7 effects are visible in diagnostics ("Effects loaded: 7").
- [ ] Hot reload works.
- [ ] Unit tests pass.
- [ ] Coverage on `EffectCatalog` ≥ 80%.
