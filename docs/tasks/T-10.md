# T-10: Channels-based event bus

- **Task ID:** T-10
- **Title:** Implement the in-process event pipeline using `System.Threading.Channels` per PRP §2.7
- **Depends on:** T-06
- **Phase:** 1
- **Estimated effort:** M (2–6 hours)
- **Branch:** `feat/10-channels-event-bus`

## Context

PRP §2.7 specifies three bounded channels (SensorEvent, GameEvent, ForceCommand) with specific capacity, FullMode, and writer/reader semantics. This task wires those channels into the host as singletons and provides the infrastructure for sensors to publish, the fusion engine to consume + publish, and the output worker to consume.

## Deliverables

1. `Moza.ScLink.Core/Bus/IEventBus.cs` interface exposing:
   - `ChannelWriter<SensorEvent> SensorEvents { get; }`
   - `ChannelReader<SensorEvent> SensorEventReader { get; }`
   - `ChannelWriter<GameEvent> GameEvents { get; }`
   - `ChannelReader<GameEvent> GameEventReader { get; }`
   - `ChannelWriter<ForceCommand> ForceCommands { get; }`
   - `ChannelReader<ForceCommand> ForceCommandReader { get; }`
2. `Moza.ScLink.Core/Bus/EventBus.cs` concrete implementation with the channel configurations:
   - SensorEvent: bounded(1024), DropOldest, SingleWriter=false, SingleReader=false
   - GameEvent: bounded(256), DropOldest, SingleWriter=true, SingleReader=false
   - ForceCommand: bounded(64), Wait, SingleWriter=true, SingleReader=true
3. `Moza.ScLink.Core/Bus/EventBusMetrics.cs` tracking:
   - Events published per channel per second
   - Drop count per channel
   - Channel depth (current count) per channel
4. DI registration: `services.AddSingleton<IEventBus, EventBus>()`.
5. A diagnostic tap pattern: multiple consumers can read the same channel by reading the `ChannelReader` and re-broadcasting. For Phase 1, only one consumer per channel is needed (fusion reads SensorEvent, resolver reads GameEvent, output worker reads ForceCommand). The metrics surface provides observability.
6. Unit tests:
   - `EventBusTests.cs` — publish-consume happy path on each channel
   - DropOldest semantics: fill SensorEvent channel beyond 1024; verify oldest drops are counted
   - Wait semantics: ForceCommand channel blocks writer when full
   - Metrics increment correctly

## Acceptance criteria

- [ ] All three channels exist with the exact `BoundedChannelOptions` from PRP §2.7.
- [ ] `EventBusMetrics` exposes drop counts and depth.
- [ ] Drop rate over 1% over a 60-second window logs a warning (implement via a periodic check in a hosted service).
- [ ] Unit tests pass.
- [ ] Coverage on `Moza.ScLink.Core.Bus` ≥ 85%.
