# T-16: Emergency stop end-to-end

- **Task ID:** T-16
- **Title:** Implement global emergency stop: hotkey + UI button → all effects halted within `EmergencyStopMaxLatencyMs` (50 ms)
- **Depends on:** T-07, T-10, T-15
- **Phase:** 1
- **Estimated effort:** M (2–6 hours)
- **Branch:** `feat/16-emergency-stop`

## Context

PRP §5.8 mandates a 50 ms emergency-stop latency budget. The stop path must bypass normal channel backpressure and reach the device synchronously. This task wires the hotkey, the UI button, and the bypass channel.

## Deliverables

1. `Moza.ScLink.Effects/EmergencyStop.cs` — exposes `Activate()` and a `Cleared` event.

   ```csharp
   public interface IEmergencyStop
   {
       bool IsActive { get; }
       event EventHandler<EmergencyStopActivatedEventArgs>? Activated;
       event EventHandler? Cleared;
       Task ActivateAsync(string reason, CancellationToken ct = default);
       Task ClearAsync(CancellationToken ct = default);
   }
   ```

2. `Moza.ScLink.Effects/ForceCommandPipeline.cs` — the worker that consumes `ForceCommands`. When `IEmergencyStop.IsActive` becomes true, it:
   - Immediately calls `device.StopAllAsync()` on the output device (bypassing the channel)
   - Drains and discards remaining commands in the channel
   - Refuses to process new `PlayEffectCommand`s until cleared
   - Allows `StopEffectCommand` and `StopAllCommand` through

3. `Moza.ScLink.App/Input/GlobalHotkey.cs` — registers a global Win32 hotkey via `RegisterHotKey` (default `Ctrl+Alt+F12`, configurable). Translates the WM_HOTKEY message via a hidden message-only window into `IEmergencyStop.ActivateAsync("hotkey")`.

4. UI: emergency-stop button on `MainWindow.xaml`. Pressing it calls `IEmergencyStop.ActivateAsync("ui")`.

5. Once activated, the UI shows a banner ("EMERGENCY STOP ACTIVE — Press Clear to resume"). The Clear button calls `IEmergencyStop.ClearAsync()`.

6. Telemetry: log the activation, the reason, and the wall-clock latency from `Activated` to last device command at `Information` level.

7. Unit tests:
   - `EmergencyStopTests.ActivateAsync_Sets_IsActive`
   - `EmergencyStopTests.ActivateAsync_Raises_Activated_Event`
   - `EmergencyStopTests.ClearAsync_Resets_State`
   - `EmergencyStopTests.Multiple_Activate_Calls_Are_Idempotent`
   - `ForceCommandPipelineTests.Discards_Play_Commands_While_EmergencyStop_Active`
   - `ForceCommandPipelineTests.Passes_Stop_Commands_Through_While_Active`
   - `ForceCommandPipelineTests.Resumes_Normal_Processing_After_Clear`
   - `ForceCommandPipelineTests.Measures_Latency_Under_50ms_With_Mock_Device`

8. Integration test: end-to-end with mock device, assert latency from `ActivateAsync` to `device.StopAllAsync()` < 50 ms over 100 iterations.

## Implementation notes

The hotkey thread must use a message-only window (`HWND_MESSAGE`) so it works while the main window is hidden. The `RegisterHotKey` call must occur on the same thread that pumps the message loop for that window.

The bypass to `device.StopAllAsync()` should not go through the `ForceCommands` channel — that would queue behind any pending commands. Instead, the pipeline reads `IEmergencyStop.IsActive` on every command and short-circuits when set. The first reading of `IsActive = true` triggers the immediate `device.StopAllAsync()`.

Configurable hotkey: store the modifier+key combination in `appsettings.json` and re-register if changed. Use `User32.RegisterHotKey` / `UnregisterHotKey` P/Invoke (these are sanctioned APIs, not on the forbidden list).

The 50 ms budget is measured from `IEmergencyStop.ActivateAsync` return-from-await to `device.StopAllAsync()` return. The hotkey-to-`ActivateAsync` latency (Win32 message pump) is not counted but should be well under 10 ms on a healthy system.
