# T-23: Phase 1 hardware validation pass

- **Task ID:** T-23
- **Title:** Run the full hardware test checklist on AB6 and AB9; tune effect parameters; sign the PR
- **Depends on:** T-07 through T-22 (everything else in Phase 1)
- **Phase:** 1
- **Estimated effort:** L (1–3 days; primarily wall-clock for soak tests)
- **Branch:** `feat/23-hardware-validation`

## Context

T-07 has its own narrow hardware-validation gate (Sections A–E of the checklist). T-23 is the comprehensive validation across the full checklist (A–H) on both supported devices. The effect parameters in `phase1-effect-catalog.json` are first-pass; this task is when they get tuned against real hardware feedback.

## Deliverables

1. `docs/hardware/T-23-checklist.md` (provided in starter; copy into the repo's `docs/hardware/` folder).
2. Completed checklists, one per device: `docs/hardware/runs/T-23-AB6-<date>.md` and `docs/hardware/runs/T-23-AB9-<date>.md`. Each fully signed.
3. Any effect parameter changes made during tuning are committed to `phase1-effect-catalog.json` as part of this PR. Each change must be justified in the PR description (e.g., "atmosphere-entry frequency lowered from 18 Hz to 14 Hz: at 18 Hz on AB9 it felt buzzy; 14 Hz feels more like atmospheric buffet").
4. Any defects discovered are filed as issues with reproduction steps and severity. Defects rated `blocker` must be fixed before this PR merges; `major` and below can be tracked into Phase 2.
5. A "Phase 1 hardware validation report" (`docs/hardware/runs/T-23-summary-<date>.md`) summarizing:
   - Devices tested
   - All effects validated
   - Final effect parameter values (with deltas from starter values)
   - Open issues (none blocker)
   - Recommendation: ship Phase 1 / hold for fixes

## Acceptance criteria

- [ ] All 8 sections of the checklist pass on AB6.
- [ ] All 8 sections of the checklist pass on AB9.
- [ ] No blocker defects open.
- [ ] Effect catalog tuned and committed.
- [ ] Summary report committed.
- [ ] PR signed by engineer, reviewer, and architect.

## Non-goals

- Not changing the architecture. Tuning is parameter-only.
- Not adding Phase 2 effects.

## Implementation notes

For the long-soak tests (Section G), schedule them on a quiet machine you can leave running. Capture Task Manager metrics every 15 minutes (a small PowerShell script can do this; not committed to the repo, just retained as evidence in the PR).

For the parameter tuning, prefer small adjustments (frequency ±2-4 Hz, intensity ±0.05, duration ±50 ms). Large changes suggest the effect concept is wrong; file an architecture issue rather than tuning past it.
