# T-07: Vortice.DirectInput output device

- **Task ID:** T-07
- **Title:** Replace hand-rolled DirectInput COM interop with Vortice.DirectInput; preserve all behaviors from the existing `DirectInputForceFeedbackDevice`
- **Depends on:** T-02 (existing code in place), T-05 (DDRs filed)
- **Phase:** 1
- **Estimated effort:** L (1–3 days)
- **Branch:** `feat/07-vortice-directinput`

## Context

The existing repository hand-marshals COM interfaces (`IDirectInput8W`, `IDirectInputDevice8W`, `IDirectInputEffect`) against `dinput8.dll`. PRP §2.3 and ADR-0001 mandate replacing this with Vortice.DirectInput. The existing code works on AB6 hardware and embodies hard-won knowledge about re-acquisition, effect caching, and HRESULT handling — every one of those behaviors must survive the migration.

This task is the single largest Phase 1 task and the one most likely to introduce regressions. The hardware test checklist (`docs/hardware/T-23-checklist.md`) gates the PR merge.

2. `Moza.ScLink.DirectInput/VorticeDirectInputAdapter.cs` — concrete implementation backed by Vortice. Wraps Vortice's `IDirectInput8`, `IDirectInputDevice8`, `IDirectInputEffect` types behind the abstraction interfaces.

3. `Moza.ScLink.DirectInput/VorticeDirectInputDevice.cs` — the new `IForceFeedbackDevice` implementation. **Mirrors every behavior of the existing `DirectInputForceFeedbackDevice`**:
   - Cooperative level: Exclusive + Background
   - Data format: Joystick
   - Two-axis Cartesian (X, Y) effects
   - Effect cache keyed by `(effectType, frequency, intensity, duration, directionX, directionY)`
   - Re-acquisition loop on `DIERR_NOTEXCLUSIVEACQUIRED` (with bounded retry: 3 attempts at 50/200/500 ms backoff before giving up)
   - Re-download on `DIERR_NOTDOWNLOADED`
   - `SetGain(deviceGain * 10000)` honors device-level gain
   - Logs each re-acquisition and each cache hit/miss at `Debug` level

4. `Moza.ScLink.DirectInput/DirectInputErrorClassifier.cs` — maps Vortice exceptions / HRESULTs to a typed enum (`Transient`, `NeedsReacquire`, `NeedsRedownload`, `Fatal`).

5. The old `DirectInputForceFeedbackDevice.cs`, `DirectInputNative.cs`, `DirectInputComInterfaces.cs`, `DirectInputStructures.cs` are deleted.

6. Unit tests against `IDirectInputAbstraction` mocks:
   - `VorticeDirectInputDeviceTests.Creates_Effect_On_First_Play`
   - `VorticeDirectInputDeviceTests.Reuses_Cached_Effect_On_Repeat_Play`

7. Manual hardware validation per the full checklist in `docs/hardware/T-23-checklist.md` (Sections A–H). PR merge is blocked until the checklist is signed by the engineer + reviewer + architect.

## Implementation notes

Vortice's `IDirectInputDevice8.CreateEffect` takes a `Vortice.DirectInput.EffectParameters` struct. The existing code's `DirectInputEffect.dwSize`, `dwTriggerButton`, `cAxes`, `rgdwAxes`, `rglDirection` translate to Vortice's typed fields directly. The two-axis Cartesian setup is:

```csharp
var parameters = new EffectParameters
{
    Flags = EffectFlags.Cartesian | EffectFlags.ObjectIds,
    Duration = durationMicroseconds,
    SamplePeriod = 0,
    Gain = 10000,
    TriggerButton = -1,  // no trigger
    TriggerRepeatInterval = 0,
    Axes = new int[] { (int)ObjectId.X, (int)ObjectId.Y },
    Directions = new int[] { directionXScaled, directionYScaled },
    Envelope = envelope,
    TypeSpecificParameters = periodicTypeSpecific
};
```

For the periodic type-specific parameters:

```csharp
var periodic = new Periodic
{
    Magnitude = (int)(intensity * 10000),
    Offset = 0,
    Phase = 0,
    Period = (int)(1_000_000.0 / frequencyHz)  // period in microseconds
};
```

When porting, keep the existing intensity-to-magnitude scaling (`intensity * 10000`) — that's the DirectInput convention and changing it would break feel parity.

For the effect-cache key, hash on the same composite as the old code so cache behavior is identical:

```csharp
private static int ComputeCacheKey(EffectType type, int frequencyHz, int magnitudeUnits, int durationMs, int dirX, int dirY)
    => HashCode.Combine(type, frequencyHz, magnitudeUnits, durationMs, dirX, dirY);
```

Re-acquisition retry pattern:

```csharp
for (int attempt = 0; attempt < 3; attempt++)
{
    try
    {
        device.Acquire();
        effect.Start(iterations, EffectPlayFlags.None);
        return;
    }
    catch (Exception ex) when (Classify(ex) == DirectInputErrorClass.NeedsReacquire)
    {
        _logger.LogDebug("DI re-acquire attempt {Attempt} after {Error}", attempt + 1, ex.Message);
        await Task.Delay(attempt switch { 0 => 50, 1 => 200, _ => 500 });
        device.Unacquire();
    }
}
_logger.LogWarning("DI re-acquire exhausted; effect not played");
```

The Vortice exception types vary by API version; the error classifier centralizes the mapping so the rest of the code doesn't need to know.
