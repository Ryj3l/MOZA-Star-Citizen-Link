# T-27: Migration convergence — host-start, canonical device registration, transitional-path removal

- **Task ID:** T-27
- **Title:** Activate the bus pipeline end-to-end: start the generic host, register the canonical force-feedback device, remove the legacy direct-drive path and transitional shims
- **Depends on:** T-07, T-08, T-11, T-12, T-14, T-15, T-16 (all merged)
- **Phase:** 1
- **Estimated effort:** L (6–12 hours)
- **Branch:** `feat/27-migration-convergence`

## Context

PRP §10.1 deliverable #9 requires the `System.Threading.Channels` event bus *"wired end-to-end (Log sensor → Fusion → Effect resolver → Output worker)"*, and §10.3 exit criteria require the app to start, enter preview mode safely without hardware, play the seven test-sequence effects on real hardware, and halt all effects within 50 ms. The §15 task decomposition (T-01–T-24, extended by the T-25/T-26 replay-harness addendum) allocated no discrete task to the step that makes this true: **start the generic host, register a canonical `Moza.ScLink.Core.Devices.IForceFeedbackDevice`, and retire the legacy device path.**

As a result, all five pipeline stages now exist and are DI-registered but **dormant**: the generic host is built-not-started (`#43`), and `ForceCommandPipeline` (T-16 PR1) could not even be registered because the canonical device is not exposed to DI — `Moza.ScLink.App/ForceFeedback/ForceFeedbackDeviceFactory` returns the *legacy* `Moza.ScLink.Core.IForceFeedbackDevice` via `LegacyForceFeedbackDeviceAdapter` wrapping a `VorticeDirectInputDevice`. This task is the gap-fill that activates the pipeline.

The gap was silent: T-17 (preview), T-18 (diagnostics), and T-16 PR2 (e-stop UI) all assume a running end-to-end stack, yet none owns starting it — and the T-16/T-17 dependency lists disagree between PRP §15 and their `docs/tasks/` specs, a symptom of no spec owning this cross-task seam. Work items are tracked in issues **#43** (start the generic host), **#45** (remove the legacy `GameLogTailer → ForceFeedbackController` direct-drive path), and **#15** (delete the transitional T-07 shims, including `LegacyForceFeedbackDeviceAdapter`).

This is *already-planned Phase-1 scope* (PRP §2.7 / §10.1 #9 / §10.3), not new capability — hence a task spec, not a PRP addendum.

## Deliverables

1. **Generic host start (#43).** The application starts the generic host so registered `IHostedService`s run (the sensor, fusion engine, effect resolver, and output worker), with correct startup/shutdown lifecycle alongside the WPF application lifetime.
2. **Canonical device registration.** Register `Moza.ScLink.Core.Devices.IForceFeedbackDevice` in DI so `ForceCommandPipeline` resolves a device: the real-hardware path is `VorticeDirectInputDevice` (canonical, unwrapped); the no-hardware path is a minimal canonical fallback (deliverable 6).
3. **`ForceCommandPipeline` hosted-service registration.** Land the registration deferred in T-16 PR1 (`AddHostedService<ForceCommandPipeline>()`), now that the canonical device resolves.
4. **Transitional shim removal (#15).** Delete `LegacyForceFeedbackDeviceAdapter` and retire the legacy `Moza.ScLink.Core.IForceFeedbackDevice` consumers that the canonical path supersedes.
5. **Legacy direct-drive removal (#45).** Remove the legacy `GameLogTailer → ForceFeedbackController` direct-drive path so the device is driven by exactly one path — the bus pipeline. Must land atomically with host-start (deliverable 1) to avoid double-driving the device.
6. **Minimal canonical no-hardware fallback device.** A canonical-interface device that lets the app start safely without hardware (§10.3) and logs intended output, until T-17's richer `PreviewForceFeedbackDevice` supersedes it.
7. **End-to-end integration test(s).** Verify the full flow (log line → sensor → fusion → resolver → safety → pipeline → device) offline with a recording/mock device, including the e-stop bypass reaching the device within the budget through the *live* pipeline.

## Acceptance criteria

- [ ] App starts on a clean Windows 11 machine without Star Citizen installed (host starts; no crash).
- [ ] App starts without any MOZA hardware connected and enters preview mode safely (the minimal canonical fallback receives commands and logs intended output).
- [ ] The Channels bus runs end-to-end (Log sensor → Fusion → Effect resolver → Output worker) with the host started — verified by an integration test.
- [ ] The device is driven by exactly one path (the bus pipeline); the legacy `GameLogTailer → ForceFeedbackController` direct-drive path is removed.
- [ ] No `LegacyForceFeedbackDeviceAdapter` or legacy `Core.IForceFeedbackDevice` references remain.
- [ ] Emergency stop halts all active effects within 50 ms through the live pipeline (the T-16 PR1 latency property holds end-to-end, not only in isolation).
- [ ] Stage 3 gates pass: Release build zero-warning, tests green, forbidden-API scan, DDR completeness, coverage thresholds.
- [ ] Hardware exit criteria (AB6/AB9 seven-effect playback + e-stop on hardware, PRP §10.3) — **Category A (hardware-gated)**, verified in the T-23 hardware validation pass; T-27 enables them.

## Out of scope

- T-17: the rich `PreviewForceFeedbackDevice` (lands after T-27, superseding the minimal fallback).
- T-18 diagnostics panel; T-19 settings persistence.

## Implementation notes

Several mechanisms are deliberately left to the execution preamble (this spec captures *what*, not *how*): the host-start integration point (WPF `OnStartup` vs `Program.Main` vs a dedicated host thread); the minimal canonical fallback device's shape (a thin canonical null device vs adapting the existing fallback chain); whether the convergence ships as one atomic PR or a sequenced set (the `#43`+`#45` atomicity requirement constrains this — the host must not start while the legacy direct-drive path is still live); and the end-to-end verification harness (how to assert the device is driven, and halts, without hardware). Preserve the PRP §14.2 behaviors that touch the device path (DirectInput re-acquisition on `DIERR_NOTEXCLUSIVEACQUIRED`, 750 ms duplicate-impact suppression, effect-cache composite keying, two-axis Exclusive+Background cooperative level) when removing the legacy path.
