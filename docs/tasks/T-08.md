# T-08: Device allowlist and detection

- **Task ID:** T-08
- **Title:** Implement device-allowlist enforcement so the app only sends force commands to MOZA AB6/AB9
- **Depends on:** T-07
- **Phase:** 1
- **Estimated effort:** S (1–2 hours)
- **Branch:** `feat/08-device-allowlist`

## Context

PRP §13.3 forbids driving unknown force-feedback devices. This task wires the allowlist from `device-allowlist.json` into the device enumerator so non-MOZA devices appear in diagnostics but receive no commands.

## Deliverables

1. `Moza.ScLink.DirectInput/DeviceAllowlist.cs` loading `device-allowlist.json` (starter file provided).
2. `Moza.ScLink.DirectInput/DeviceDetector.cs` that enumerates DirectInput FFB devices, matches each against the allowlist, assigns a `DeviceModel`, and returns a list of `(deviceInfo, deviceModel)` tuples.
3. Devices not in the allowlist appear in `DeviceModel.Unknown` and are listed in diagnostics but never receive `ExecuteAsync` commands. The `VorticeDirectInputDevice` constructor refuses to be created for `DeviceModel.Unknown` (throws or returns null from a factory method).
4. Unit tests:
   - `DeviceAllowlistTests.cs` — match a synthetic "MOZA AB6 FFB Base" product name against the allowlist (positive)
   - Match "Logitech G27" (negative)
   - Match case-insensitively
   - Match with partial names ("AB6")
   - Handle empty product name gracefully (returns Unknown)

## Acceptance criteria

- [ ] `device-allowlist.json` loaded on app startup.
- [ ] DirectInput enumeration produces a list including detected MOZA flight base(s) and any other FFB devices.
- [ ] MOZA AB6 → `DeviceModel.MozaAb6`. MOZA AB9 → `DeviceModel.MozaAb9`. Anything else → `DeviceModel.Unknown`.
- [ ] `VorticeDirectInputDevice` cannot be constructed for `Unknown`.
- [ ] Diagnostics panel lists all enumerated devices with their assigned model.
- [ ] Manual test: connect a non-MOZA controller; verify it appears in diagnostics but the Test Sequence button does not drive it.
- [ ] Unit tests pass; coverage ≥ 70% on this code.
