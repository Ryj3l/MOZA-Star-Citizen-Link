# T-18: Diagnostics panel

- **Task ID:** T-18
- **Title:** Implement the diagnostics panel that surfaces sensor health, device state, channel metrics, rule metrics, recent commands, and recent log lines
- **Depends on:** T-09 (MVVM), T-10 (Bus), T-11, T-12, T-16, T-17
- **Phase:** 1
- **Estimated effort:** M (2–6 hours)
- **Branch:** `feat/18-diagnostics-panel`

## Context

The existing `ForceFeedbackDiagnostics` and `MozaSdkDiagnosticsViewModel` are starting points. PRP §10 mandates a unified diagnostics surface so beta users can self-triage and engineers can debug without log diving.

## Deliverables

1. `Moza.ScLink.Diagnostics/DiagnosticsViewModel.cs` aggregating signals from:
   - `IEventBus` (channel depth, drop counts, throughput per channel)
   - `IDeviceState` (current device, connection state, last error)
   - `IFusionEngine` (rules loaded, firings per rule per minute, suppressed counts)
   - `IEffectCatalog` (effects loaded, last reload time)
   - `IPatternLibrary` (patterns loaded by file, last reload time)
   - `ISensor` health for each registered sensor
   - Recent `ForceCommand` ring buffer (last 100)
   - Recent log lines published by sensors (last 100)
2. UI panel layout (`DiagnosticsPanel.xaml`):
   - **Header:** product version, build SHA, runtime info (OS, .NET version)
   - **Sensors:** status, events emitted, drops, last event time per sensor
   - **Channels:** depth, drops/sec, throughput per channel
   - **Fusion:** per-rule firing counts and suppression counts
   - **Device:** current device model, state, last command time, active effects count
   - **Recent commands:** last 20 with timestamps and intensities
   - **Recent log events:** last 20 raw matched lines
3. Refresh rate: 4 Hz (250 ms timer). Lower refresh rates for static fields.
4. Export button: produces a redacted diagnostic bundle (T-19 owns redaction; the button calls into the redactor).
5. Unit tests:
   - `DiagnosticsViewModelTests.Aggregates_All_Sources`
   - `DiagnosticsViewModelTests.Bounds_Recent_Commands_Buffer_At_100`

## Acceptance criteria

- [ ] All listed signals visible in the UI.
- [ ] Refresh is smooth (no UI jank).
- [ ] Export button produces a zip file at a user-selected path.
- [ ] Coverage on `Moza.ScLink.Diagnostics` ≥ 70%.
- [ ] Hardware test checklist Section H signed off (diagnostic capture and redaction verified).

## Implementation notes

Use `CommunityToolkit.Mvvm`'s `IRecipient<TMessage>` pattern for cross-thread signal delivery from sensors/engines into the view model, or use `IObservable<T>` plus `ObserveOn(DispatcherScheduler.Current)`. Whichever you pick, keep the UI bind path off the hot path of the data pipeline.

Channel depth introspection requires that `EventBus` expose read-only properties. `Channel<T>` does not expose count directly; track count via Interlocked counters incremented on write and decremented on read.
