# T-12: Fusion engine with rule loader

- **Task ID:** T-12
- **Title:** Implement the fusion engine that consumes `SensorEvent` and emits `GameEvent`, with rules loaded from JSON
- **Depends on:** T-06, T-10, T-11
- **Phase:** 1
- **Estimated effort:** L (1–3 days)
- **Branch:** `feat/12-fusion-engine`

## Context

PRP §6 specifies a rule-based fusion engine that correlates evidence from multiple sensors and emits confirmed `GameEvent`s. In Phase 1, only the log sensor produces evidence, so most rules degenerate to single-sensor confirmations. The architecture, however, must be ready to incorporate audio and screen sensors in Phase 2 without restructuring.

The fusion engine is the brain of the application. It owns the rule library, the evidence-buffering window, and the de-duplication policy.

## Deliverables

1. `Moza.ScLink.Fusion/FusionEngine.cs` — hosted background service:
   - Reads from `IEventBus.SensorEventReader`
   - Maintains a sliding evidence buffer per `(GameEventType, primary entity)` key, with a configurable window (default 750 ms — preserves the existing duplicate-impact suppression behavior)
   - Evaluates loaded rules against the buffer on every new sensor event
   - When a rule fires, publishes a `GameEvent` to `IEventBus.GameEvents` and clears the relevant buffer entries
   - Tracks per-rule metrics: firings/sec, confirm-vs-suppress counts

2. `Moza.ScLink.Fusion/Rules/FusionRule.cs` — record describing a rule:

   ```csharp
   public sealed record FusionRule(
       string RuleId,
       GameEventType ProducesEventType,
       string Description,
       IReadOnlyList<EvidenceRequirement> Requirements,
       TimeSpan EvidenceWindow,
       double MinConfidence,
       string SuppressionKey);

   public sealed record EvidenceRequirement(
       SensorKind Kind,
       string EventType,
       double Weight,
       bool Required);
   ```

3. `Moza.ScLink.Fusion/Rules/RuleLibrary.cs` — JSON loader, hot-reload via `FileSystemWatcher`, schema versioning.

5. Suppression policy: when a rule fires, no rule with the same `suppressionKey` may fire again within `evidenceWindowMs` (preserves the existing landing-impact debounce).

6. Confidence computation: sum of (matched-requirement weights) / (total required weight). Phase 1 rules are single-sensor so confidence is 0.0 or 1.0; Phase 2 will use partial confidence.

## Acceptance criteria

- [ ] All 7 Phase 1 rules fire correctly given synthetic sensor events.
- [ ] Existing 750 ms landing-impact debounce preserved (verified via test).
- [ ] Hot reload of `phase1-rules.json` works.
- [ ] Per-rule metrics visible in diagnostics panel.
- [ ] Coverage on `Moza.ScLink.Fusion` ≥ 85%.
- [ ] Manual end-to-end test: feeding a Game.log sample (or replaying real Star Citizen events) produces the expected sequence of `GameEvent`s, which produce expected `ForceCommand`s, which play on hardware.

## Implementation notes

The evidence buffer can be a simple `Dictionary<string, List<(SensorEvent evt, DateTimeOffset receivedAt)>>` keyed by `(GameEventType, suppressionKey)` with periodic cleanup of entries older than the longest configured window.

For Phase 2 extensibility, design the `EvidenceRequirement` evaluator as a plug-in lookup so audio-frequency requirements and screen-pixel requirements can be added without restructuring.
