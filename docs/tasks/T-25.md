# T-25: Light replay CLI

- **Task ID:** T-25
- **Title:** Implement a standalone CLI replayer that reads a bundle, replays its Game.log through the pattern library, and emits a structured event stream. Includes manifest validator and bundle stripper.
- **Depends on:** T-06 (core domain types), T-11 (log sensor + pattern library)
- **Phase:** 1-parallel (can run alongside Phase 1 once T-11 merges)
- **Estimated effort:** M (2–6 hours)
- **Branch:** `feat/25-replay-cli-light`

## Context

PRP addendum `docs/PRP-ADDENDUM-replay-harness.md` introduces a development-only replay harness. T-25 ships the light CLI: a console app that reads a bundle, parses the Game.log through the existing pattern library, and writes a `events.jsonl` stream. It is the foundation T-26 builds on.

T-25 deliberately does **not** wire into the event bus, fusion engine, or resolver — that's T-26. The light CLI's job is to make pattern-library iteration fast: edit a regex, re-run the CLI against a saved bundle, see which lines matched. No UI, no hardware, no audio, no screen.

The bundle format is specified in `docs/replay-bundle-schema.md` (schema version 1). T-25 implements the manifest reader and the Game.log path. Audio and screen feature files are recognized by the validator but otherwise ignored at this stage.

## Deliverables

1. **`tools/Moza.ScLink.Replay/`** — new .NET 8 console project under `tools/` (not `src/`). Csproj sets `OutputType=Exe`, references `Moza.ScLink.Core` (for `SensorEvent`/`GameEvent`/`ScEventKind`) and `Moza.ScLink.Logs` (for the pattern library and `StarCitizenEventParser`).
2. **`tools/Moza.ScLink.Replay/Bundle/BundleManifest.cs`** — record type matching the manifest schema. Uses `System.Text.Json` with source generators for AOT-readiness.
3. **`tools/Moza.ScLink.Replay/Bundle/BundleReader.cs`** — loads a manifest, validates schema version, opens the Game.log file at the recorded byte range. For T-25, audio/screen sections are read but not validated structurally (T-26 finishes them).
4. **`tools/Moza.ScLink.Replay/Bundle/BundleValidator.cs`** — implements the validation rules from `docs/replay-bundle-schema.md` § Validation rules. For T-25, audio/screen line-count rules are advisory warnings only (not failures) until T-26 wires audio/screen.
5. **`tools/Moza.ScLink.Replay/Bundle/BundleStripper.cs`** — implements stripping. For T-25, that means: copy manifest with `stripped: true`, copy `game.log`, copy `audio-features.jsonl` and `screen-features.jsonl` if present, copy `README.md`, optionally `screen-snapshots/`, refuse to overwrite without `--force`, write `STRIPPED.txt` provenance marker. Does **not** copy `audio.wav` or `screen-rois.mp4`.
6. **`tools/Moza.ScLink.Replay/Program.cs`** — command-line entry point with three subcommands:
   - `replay <bundle-path> [--out <dir>]` — validate, then iterate Game.log lines through the pattern library, write `events.jsonl` and `replay-report.md` to `<out>` (default: `<bundle-path>/replay-out/<timestamp>/`).
   - `validate <bundle-path>` — run validation, print pass/fail report, exit 0 or 1.
   - `strip <rich-bundle> <stripped-output> [--keep-snapshots] [--force]` — produce a feature-only copy.
   Use `System.CommandLine` (already-pre-approved dependency? — confirm against `Directory.Packages.props`; if not, file a one-line DDR for it).
7. **`tools/Moza.ScLink.Replay/Output/EventJsonl.cs`** — writes one JSON line per detected event. Schema:
   ```json
   {"tMs": 12453, "kind": "QuantumSpoolStarted", "name": "Quantum spool vibration", "patternId": "log.quantum_spool_start", "matchedLine": "<verbatim>", "intensity": 0.42, "durationMs": 8000}
   ```
8. **`tools/Moza.ScLink.Replay/Output/ReplayReport.cs`** — writes a human-readable `replay-report.md` summarizing: bundle metadata, validation result, total events emitted by kind, top-N most-frequent matched patterns, top-N unmatched lines (sampled), suggested pattern coverage stats.
9. **`tools/Moza.ScLink.Replay.Tests/`** — xUnit test project:
   - `BundleManifestTests` — round-trip serialization, schema-version rejection
   - `BundleValidatorTests` — happy path, missing files, line-count mismatch
   - `BundleStripperTests` — produces correct artifacts, refuses overwrite without `--force`, never copies raw audio/video
   - `ReplayProgramTests` — end-to-end against a fixture bundle (created in `Fixtures/`); asserts `events.jsonl` matches an expected golden file
10. **`tools/Moza.ScLink.Replay.Tests/Fixtures/sample-bundle/`** — a minimal hand-crafted bundle: 3-line `game.log` containing one quantum-spool-start line, a minimal `manifest.json`, no audio or screen. Used by the program tests.
11. **`tools/Moza.ScLink.Replay/README.md`** — usage examples and CLI reference. Linked from the main `README.md` § Development.
12. **Solution wiring** — the new projects are added to `MozaStarCitizenLink.sln`. The tools projects are excluded from the App's release publish via `<IsPackable>false</IsPackable>` and explicit project-reference exclusion in `Moza.ScLink.App.csproj`.
13. **`.gitignore`** — add `tools/recordings/` to the repo's existing `.gitignore` (the kit's `.gitignore` already has this; T-25 confirms it).

## Non-goals

- No audio replay (T-26).
- No screen replay (T-26).
- No event bus / fusion / resolver wiring (T-26).
- No hardware playback (future task, requires hard guards).
- No recorder — T-25 consumes bundles, it does not produce them. Hand-assembled bundles are sufficient for development at this stage.
- No CI fixture-replay regression test — that comes after T-26.
- No `--speed Nx` faster-than-real-time flag — light CLI iterates the Game.log as fast as it can already; speed flags only matter once audio/screen replay exists in T-26.

## Implementation notes

- The tools directory pattern: keep all dev-only tooling under `tools/` so a `dotnet publish` of the App can confidently exclude it. The CI workflow may want a separate `tools-build` job that builds and tests the tools projects without including them in the publish artifact.
- `System.CommandLine` is in preview for .NET 8 but is the cleanest option. Alternative: a thin custom argument parser. Either is fine; pick one and stick with it.
- The replay's pattern matching must use **exactly** the same `StarCitizenEventParser` instance as the live log sensor — same regex compilation, same timeout, same `unsupported: true` skip logic. The whole point of replay is to reproduce live behavior; any divergence defeats the purpose.
- Replay does not need a real `ISensor`; T-25 reads `game.log` directly and calls the parser. T-26 will wrap this in a `ReplayLogSensor : ISensor`.
- The `events.jsonl` schema is intentionally narrower than `SensorEvent` because T-25 doesn't go through the full sensor model. T-26's output schema will be the wider `SensorEvent` / `GameEvent` shape.
- Write the report in Markdown, not HTML. Devs read it in VS Code or on GitHub.
- For the sample fixture bundle's `game.log`: use real (or realistic) Star Citizen quantum-spool log lines. If the pattern library is still using placeholder regexes from `v0.json` (as expected through Phase 1), the fixture needs at least one line the placeholder pattern matches; pick a line that's robust to pattern improvements.

## Operator note

After T-25 merges, the workflow for tuning a pattern becomes:

1. Operator hand-assembles or imports a bundle into `tools/recordings/`.
2. `dotnet run --project tools/Moza.ScLink.Replay -- replay tools/recordings/<bundle>`.
3. Inspect `events.jsonl` and `replay-report.md`.
4. Edit `Moza.ScLink.Logs/Patterns/v0.json`.
5. Re-run step 2. Compare. Iterate.

No Star Citizen needed. No wheelbase needed. Single-machine, fully offline.
