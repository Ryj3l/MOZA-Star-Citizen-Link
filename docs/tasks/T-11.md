# T-11: Log sensor refactor to ISensor

- **Task ID:** T-11
- **Title:** Refactor `GameLogTailer` + `StarCitizenEventParser` into an `ISensor` implementation publishing `SensorEvent` to the event bus
- **Depends on:** T-06 (Core types), T-10 (Event bus)
- **Phase:** 1
- **Estimated effort:** M (2–6 hours)
- **Branch:** `feat/11-log-sensor`

## Context

Existing code tails Game.log and raises an event with the parsed `ScGameEvent`. PRP §5.3 establishes the `ISensor` interface; this task refactors the log path to conform.

The pattern library is versioned per PRP §7. The Phase 1 `v0.json` contains placeholder patterns from the existing code, plus a flag `"unsupported": true` on patterns that have not been validated against real Star Citizen log captures. Unsupported patterns are loaded but produce no events.

## Deliverables

1. `Moza.ScLink.Logs/LogSensor.cs` implementing `ISensor` per §5.3:
   - SensorId: `"log.game-log"`
   - Kind: `SensorKind.Log`
   - `StartAsync` opens `GameLogTailer` and begins reading
   - Each parsed line yields a `SensorEvent` with `EventType` derived from the matched pattern's kind
   - `StopAsync` cleanly shuts down the tailer
2. `Moza.ScLink.Logs/Patterns/PatternLibrary.cs` loading versioned pattern JSON files from `Moza.ScLink.Logs/Patterns/*.json` with file watcher for hot reload.
3. `v0.json` patterns annotated with `"unsupported": true` for placeholders. A pattern with this flag is loaded, compiled (for regex validity), but the sensor never emits events for it.
4. Preserve all existing behaviors: truncation detection, byte-position tracking, single-instance file access via `FileShare.ReadWrite | FileShare.Delete`.
5. Map old `ScEventKind` → new `SensorEvent.EventType` strings (e.g., `ScEventKind.LandingImpact` → `"log.landing_impact_candidate"`).
6. Unit tests:
   - Sensor lifecycle (Start → events flowing → Stop)
   - Pattern compilation including regex timeout protection (100ms)
   - Truncation handling
   - Hot reload of patterns (modify the JSON file, verify recompilation within 1 second)
   - Malformed JSON: prior good pattern set remains in effect
   - Unsupported patterns do not emit events

## Acceptance criteria

- [ ] Log sensor publishes `SensorEvent` instances to `IEventBus.SensorEvents`.
- [ ] Sensor health surface reports running/faulted state, events-emitted count, dropped-events count.
- [ ] Hot reload works (verified by editing `v0.json` while app is running and observing log).
- [ ] Regex timeout enforced at 100 ms — a deliberately catastrophic pattern fails compilation cleanly.
- [ ] Unit tests pass.
- [ ] Coverage on `Moza.ScLink.Logs` ≥ 80%.
- [ ] Manual test: app still detects events from a sample Game.log fixture.

## Implementation notes

The existing `StarCitizenEventParser.ResolveIntensity` logic that extracts relative-velocity magnitude from collision lines is valuable — preserve it as a feature extractor that populates `SensorEvent.Features["relativeVelocityMagnitude"]`. Fusion can use this feature in confidence boosting later.

The pattern library JSON format adds three fields versus the existing `EventPattern`:

```json
{
  "kind": "LandingImpact",
  "name": "Landing/impact bump",
  "pattern": "...",
  "intensity": 0.75,
  "durationMs": 260,
  "unsupported": false,
  "minStarCitizenBuildVersion": "3.24.0",
  "maxStarCitizenBuildVersion": null
}
```

For Phase 1, `minStarCitizenBuildVersion` and `maxStarCitizenBuildVersion` are advisory (logged in diagnostics) but not enforced. Phase 2 task will read the build version from log header and gate patterns by version range.
