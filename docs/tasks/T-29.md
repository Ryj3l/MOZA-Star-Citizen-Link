## Context

The project's standing rules are enforced today by a layered surface: BannedApiAnalyzers +
`BannedSymbols.txt` (managed API bans), `scripts/forbidden-api-scan.ps1` (regex over
.cs/.xaml/.csproj incl. string literals), `scripts/check-ddr-completeness.ps1` (dependency
governance), `.editorconfig` + TWAE/WL5 (style + quality), and review discipline. The
discovery inventory (2026-06-11) found two compile-time upgrades worth making and several
not worth making.

Key inventory facts:
- The codebase hand-rolls `LoggerMessage.Define` static `Log` classes — 34 `EventId` sites
  across 8 files — while `dotnet_diagnostic.CA1848.severity = none`: the convention exists
  only by discipline. .NET 8's built-in `[LoggerMessage]` generator deletes the boilerplate
  and enforces per-class EventId uniqueness (SYSLIB1006) with zero new packages.
- The forbidden-API scan's code-pattern leg needed comment-blanking workarounds
  (`forbidden-api-scan.ps1:108-118`) that a semantic analyzer makes unnecessary; the script's
  non-code surface (csproj version pins, xaml, string-literal smuggling) is NOT replaceable
  by an analyzer and is retained.
- Layering is already compile-enforced by the ProjectReference graph (Core→∅;
  DirectInput/Effects/Fusion/Logs→Core; Diagnostics→Core+DirectInput; App→all) — a layering
  analyzer was evaluated and DECLINED.

## Milestone 1 — [LoggerMessage] generator migration (zero new dependencies)

1. Migrate every hand-rolled `LoggerMessage.Define` static `Log` class to `[LoggerMessage]`
   partial methods (8 files: DropRateMonitor, SafetyLimiter, GlobalHotkey,
   ForceCommandPipeline, EmergencyStop, PreviewForceFeedbackDevice, VorticeDirectInputDevice,
   DeviceInitializer — re-enumerate at execution; the count may have drifted).
2. **OUTPUT-PRESERVATION CRITERION (supervisor amendment a, non-negotiable):** every existing
   EventId and message template is preserved EXACTLY at the rendered-output level. The logs
   are the forensic instrument and the historical grep surface (e.g. EventId 12's
   "DI StopAll: device contended…", "Emergency stop ACTIVATED (reason: {Reason})") must
   survive the migration byte-identical. Any unavoidable deviation is surfaced individually
   before commit — not batched, not silently adapted.
3. Re-enable CA1848 (remove `severity = none` from `src/.editorconfig`) so regression to raw
   `ILogger` calls fails the build. Pre-verify whether any raw calls exist (inventory
   uncertainty — unverified); if found, surface before migrating them.
4. CLAUDE.md Patterns section: add the `[LoggerMessage]` convention line (replaces the
   `LoggerMessage.Define`-backed guidance in the dependencies-demonstration pattern).
5. C3 policy doc (settled at spec approval): CLAUDE.md Style section gains one line — test
   names are PascalCase, no underscores; CA1707 enforced, no test-scoped suppression. (Three
   prior arcs conformed: SafetyLimiterTests, PreviewForceFeedbackDeviceTests, T-28 catalog
   tests.)
6. Rider: fix the comment drift at `src/BannedSymbols.txt:21-24` — "the custom Roslyn
   analyzer" → "the forbidden-api scan script" (one line; the comment mislabels the regex
   script as an analyzer).
7. Empirical-verification probes at execution (probe-by-compile before dependent code):
   SYSLIB1006's exact duplicate-EventId enforcement shape; generator behavior with the
   existing `EventId(n, name)` two-part construction.

## Milestone 2 — analyzer project + MSL0001 (forbidden P/Invoke)

1. New projects: `src/analyzers/Moza.ScLink.Analyzers` (netstandard2.0 — Roslyn requirement)
   + `tests/Moza.ScLink.Analyzers.Tests`.
2. **MSL0001** (severity: error): reports any `[DllImport]`/`[LibraryImport]` whose resolved
   entry-point name matches the banned native API list (OpenProcess, ReadProcessMemory,
   WriteProcessMemory, VirtualAllocEx, CreateRemoteThread, NtCreateThreadEx, NtOpenProcess,
   ZwOpenProcess, SetWindowsHookEx — keep in lockstep with `forbidden-api-scan.ps1`).
   Semantic attribute inspection — immune to the comment/string-literal false-positive class
   the script works around. No codefix (there is no fix; the API is forbidden).
3. Wiring: consumed via `ProjectReference` with `OutputItemType="Analyzer"`
   `ReferenceOutputAssembly="false"` added in `src/Directory.Build.props` with a
   self-exclusion condition (the analyzer project must not reference itself). No NuGet
   packaging — local consumption only. Probe the wiring by compile before rollout.
4. New packages (build-time, PrivateAssets=all) — NONE pre-approved, DDRs required per the
   §4.2 rule and `check-ddr-completeness.ps1` (which scans all csproj):
   `ddr-microsoft-codeanalysis-csharp.md`, `ddr-microsoft-codeanalysis-analyzers.md`, plus
   the analyzer-testing package DDR for the test project (exact package chosen at execution;
   probe the current Microsoft.CodeAnalysis.Testing package shape first).
5. **Authority footprint (supervisor amendment b):** ADR-0002 ("In-repo Roslyn analyzers for
   guardrail enforcement" — placement, wiring, script-coexistence policy, one-way-ratchet
   extended to MSL diagnostics) + the DDRs + a ONE-LINE addition to the PRP §3 layout table
   for `src/analyzers/` (surfaced like any PRP edit — the sovereign doc's normative table
   must not go stale). NO addendum document.
6. Script relationship: belt-and-suspenders. The scan script retains its full pattern list;
   slim its code-pattern leg only after one release of proven MSL0001 parity. CI unchanged
   (analyzers ride `dotnet build`).
