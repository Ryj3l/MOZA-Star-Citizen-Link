### Diagnostic narrative — A3/A4/A5 hot-plug architecturally absent (#71)

A3 was exercised on 2026-05-29 with AB9 unplugged at app launch (preview-mode baseline confirmed at `2026-05-29T09:21:21` — `Device state changed: Disconnected -> Ready (No hardware (preview mode))`). At `2026-05-29T09:22:20.0280906-05:00` the grep baseline was pinned and the operator plugged AB9 in. **No log entry followed for 90+ seconds** — the app received nothing from the plug-in, the UI stayed in preview, no exception was raised.

Root-cause empirical evidence (full diagnosis in #71):
- Windows recognizes the AB9: `Get-PnpDevice` shows VID_346E/PID_1000 devices in OK status (HID game controller × 2, USB Serial COM6, USB Composite). The plug succeeded at the OS level.
- App has **no WM_DEVICECHANGE hook** — grep for `WndProc|WM_DEVICE|HwndSource|AddHook` in `src/Moza.ScLink.App` finds only `Input/GlobalHotkey.cs` (Ctrl+Alt+F12 from T-16 PR2).
- App has **no `RegisterDeviceNotification` call** anywhere in `src/`.
- `IDeviceAvailabilityObserver` interface exists in Core but has **zero implementations** in current build.
- `ForceFeedbackDeviceFactory.cs:37` documents the deferral verbatim: "Decides ONCE at startup. … Canonical hot-plug + an SDK-bridge tier on this interface are tracked as a follow-up (PRP §10.1, line 1011)." (Citation drift — line 1011 is in §12.4, not §10.1; folded into #68.)

By architectural symmetry, A4 (hot-unplug while in canonical Ready) and A5 (replug after A4) are blocked by the same absence — the canonical device has no observer wiring.

M14 evidence (post-Pass-2 V9, PIDs 36964/40592, 2026-05-18, `docs/hardware/T-23-checklist.md`) establishes A3/A4/A5 all PASSED in M14 via the `MainWindow.OnWindowMessage` → `FallbackForceFeedbackDevice` observer pipeline that T-27 implementation PR #61 retired without replacement.
