### Section C (emergency stop)

#### Methodology (recorded before the C1/C3 runs — build `ded58cf`)

**E-stop bypass — architectural characterization (read-only, from `ForceCommandPipeline.cs`, build `ded58cf`).**
The emergency-stop bypass is **wake-immediate but execution-non-preemptive**:

- **Wake (immediate, load-independent):** `OnEmergencyStopActivated` captures the activation timestamp synchronously (`Stopwatch.GetTimestamp` via `Volatile.Write`) and cancels `_estopTrigger`. That cancels the linked token the worker is parked on, so an **idle** `WaitToReadAsync` throws `OperationCanceledException` at once → `woken = true`.
- **Execution (non-preemptive):** the actual `StopAllAsync` runs only at the **top of the loop iteration** (`RunEmergencyStopBypassAsync`). It does **not** interrupt an in-flight `ProcessAsync` / `_device.ExecuteAsync`, nor the inner drain of an already-readable queue. If e-stop fires mid-command, the bypass waits for that command (and the remaining readable queue) to finish before `StopAll` issues.
- **Consequence for the metric:** measured latency = (activation timestamp → `StopAll` complete) and is therefore **conditional on the in-flight command duration + queue depth at activation time**. An idle/sustained-playing measurement characterizes the fast path **only** and must not be reported alone as the safety contract.

**Metric scope (what the number includes / excludes).** The recorded latency is **bypass-internal**: it starts at the activation timestamp set *inside* `OnEmergencyStopActivated`. It therefore **excludes input dispatch** — the UI-click handler or the global-hotkey hook → `OnEmergencyStopActivated` leg — which is additional to true press→halt and is normally small. **C2** is where that hotkey-dispatch leg is confirmed non-laggy; C2 also captures its **own** bypass latency, cross-checked against the C1/C3 fast-path baseline (path-independence: UI-click vs hotkey should land in the same band).

**C3 — two-path latency protocol (BOTH paths required).**

1. **Fast path (typical case).** E-stop while idle or while a sustained effect is already playing (loop parked in `WaitToReadAsync`; device plays autonomously). ≥20 trials: press → wait 1s → Clear → wait 1s → repeat. Grep `EmergencyStopBypass` "(X.XX ms from activation)". p95 = sorted index `ceil(0.95·N) − 1`. **Expand the run if p95 > ~40ms OR any single sample exceeds 50ms.**
2. **Load path (REQUIRED — the only contract-under-stress probe).** A playing *sustained* effect does **not** exercise this path (it leaves the loop idle = fast path); the only probe is e-stop fired *during command processing / a draining burst*.
   - **a. Burst trials:** rapid Test-Quantum burst → immediate e-stop. Per trial, verify from the log whether the bypass **caught the drain** (latency elevated vs the fast-path baseline). **Only drain-catching trials count as load-path samples** — manual timing may miss the small non-preemptive window.
   - **b. Analytical worst-case bound (theoretical ceiling).** Drain depth is hard-bounded by the **ForceCommand channel capacity = 64** (`EventBus.cs:45` — `BoundedChannelOptions(64)`, `FullMode = Wait`; the resolver backpressures rather than overflowing, so the bypass can never wait behind > 64 commands). Theoretical worst-case = 64 × per-command processing time, where per-command time is taken from the gap between consecutive pipeline log lines in the diagnostic log. This is **conservative**: (i) inter-line gaps include scheduling/logging overhead beyond pure processing, and (ii) while e-stop is active, queued `PlayEffectCommand`s are *refused* (log-only, no device call), so a real 64-deep drain is far cheaper than 64 full `ExecuteAsync` calls — the bound over-estimates in the safe direction.
   - **c. Realistic max-burst-depth (reachability — severity framing).** The pipeline has **no event→command amplification**: Fusion emits exactly one `GameEvent` per triggering `SensorEvent` (`FusionEngine.cs:179-191`, single `TryWrite`); the resolver emits a small per-event command set — typically 1 (occasionally a stop+play) — each `WriteAsync`'d with Wait backpressure at 64 (`EffectResolverService.cs:56-58`). So realistic ForceCommand production ≈ the game-event rate (~1/event), which in Phase-1 is gameplay-paced (seconds apart) vs a per-command device drain of ~ms → **realistic queue depth ≈ 0–1**. The 64 ceiling is only reachable by deliberate flooding (button-mash burst or replay-harness log flood), **not normal Phase-1 play**.
3. **Judgment (theoretical gates the contract; realistic frames the severity).**
   - The safety contract must hold at the **permitted worst case**: if **theoretical-worst (64 × per-command) > 50ms** → **FINDING regardless** — *non-preemptive e-stop, latency unbounded by in-flight drain duration*. (This is the **anticipated** outcome given per-command device-I/O.)
   - The **realistic max-burst-depth** sets the closeout severity framing: *latent low-reachability gap* (queue depth ≈ 0–1 in Phase-1; the worst case requires deliberate flooding) **vs** *active risk* (reachable in normal play). The fast-path p95 must independently be **< 50ms**.
   - **Route to T-23 closeout** as a Phase-1 architect input (alongside #71 / inverted-shape) **with both numbers**: the theoretical worst-case (contract-violation check) and the realistic depth (reachability/severity).
